核心任务

- 创建技术计划：用简洁的语言描述用户想要构建的功能。
- 研究相关文件：找出需要修改或创建的文件和函数。
- 避免非技术内容：不包含任何产品经理风格的内容，如成功标准、时间表、迁移方案等。
- 不写实际代码：计划中不应包含具体的代码实现。
- 包含具体细节：必须包含用户提示(包括PRDs中的PRD.md)中的具体和逐字细节，以确保计划的准确性。

计划结构

- 上下文描述：在顶部提供一个简短的描述，设定背景。
- 相关文件列表：明确指出所有相关的文件和函数。
- 算法解释：逐步解释所使用的算法。
- 分阶段实施：如果需要，将工作分解为逻辑阶段。理想情况下，应先有一个“数据层”阶段，定义类型和数据库变更，然后是多个可以并行进行的阶段（例如，Phase 2A - UI, Phase 2B - API）。只有在功能非常大时才需要分阶段。

特殊情况处理

- 如果用户需求不明确，尤其是在研究了相关文件后，可以在编写计划前最多提出5个澄清问题。如果这样做了，需要将用户的回答纳入计划中。
- 优先保持简洁和精确，使计划尽可能紧凑，同时不丢失用户需求的任何关键细节。

输出规范

- 将计划写入 N_[PLAN.md](http://PLAN.md) 文件，其中 N 是下一个可用的功能编号。
